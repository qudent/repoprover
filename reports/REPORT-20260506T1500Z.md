Representative rules from the latest transitive-localdeps payload:

```text
Do not infer or reveal hidden target Lean declaration names for the selected unit.
Do not write theorem/lemma Lean code in target_statement_sketch; exact API syntax belongs in needed_mathlib_context and will be hydrated by tools.
When a source theorem states a result for several cases such as upper/lower, left/right, or an `or` hypothesis, prefer separate narrow planned_declarations for each case when the checked APIs or natural Lean statements are case-specific. Do not collapse case-specific source claims into one disjunctive theorem just to reduce task count.
If a broad source unit needs a construction or auxiliary lemma before the main claim can be stated/proved, split it into ordered planned_declarations with role same_unit_helper followed by role main_claim and explicit depends_on_task_ids.
Same-unit helper tasks are declarations for the later generator to create from the selected source unit, not prior context. Do not use hidden aligned/referencing Lean names for them.
Use previous project declarations only if they are shown under prior_project_context.
Use local_file_predecessor_declarations only as same-file helper/style context; the selected unit's aligned/referencing target declarations are omitted.
Do not treat Mathlib as the only context; enumerate source/project/local/Mathlib context separately.
Prefer exact Mathlib names when known; otherwise give a narrow query plus expected signature shape.
Keep added context tight: prefer a few thousand tokens or less per source unit.
```

The payload layout is: `rules`, `schema`, `task`, then unit records containing
verbatim source units, source-context candidates, prior project context, local

### Generation

Source: `scripts/run_latex_statement_generation.py`.

System message:

```text
You are a Lean 4 autoformalization agent. Generate a small ordered sequence of Lean declarations for the provided LaTeX theorem-like unit. Use only the source text, selector plan, previous-project context if shown, and Lean-checked Mathlib signatures in the prompt. The original aligned Lean declarations, names, statements, and proofs are withheld. Return exactly one JSON object.
```

Representative current instructions:

```text
Do not use sorry, admit, placeholders, or comments standing in for proof.
Do not include import statements or markdown fences.
Do not ask for or infer the hidden aligned Lean declaration names/statements/proofs.
Treat selector_unchecked_statement_sketch as non-authoritative mathematical intent only; do not copy its Lean syntax verbatim.
Follow the Lean-checked signatures exactly when they differ from the selector's expected shape.
Do not enlarge a finite source choice space to all functions into an infinite codomain. Before writing a theorem that sums/products over `alpha -> beta` or uses a `Fintype (alpha -> beta)` instance, check that visible context or Mathlib supplies `Fintype beta`; there is no such generic instance for infinite codomains such as `Int`. If the source is about finite signs, subsets, bounded tuples, or another finite choice space, preserve or define that finite representation instead of replacing it by all functions into an infinite type.
When planned_declarations contains role same_unit_helper, generate that helper declaration before any main_claim that lists it in depends_on_task_ids.
Same-unit helper declarations are new declarations to create from the visible source unit. You may choose fresh descriptive names for them, but do not reuse or guess hidden aligned Lean names.
If you cannot produce a complete proof from visible context, set status to cannot_prove_from_visible_context and leave lean_file_body empty. Never output Lean containing sorry/admit/placeholders.
When status is cannot_prove_from_visible_context, do not use lean_file_body as a scratchpad. Put analysis in notes only; lean_file_body must be exactly "" and declaration_names exactly [].
```

## Did Progress Go In Circles?

Mostly forward on infrastructure and diagnosis; partially circular on model
acceptance.

Forward signs:

- dev panel improved from `2/5` to `3/5` after targeted proof-lane acceptance;
- fresh-slice failures became cleaner and more auditable;
- invalid JSON and placeholder skeletons are now handled by contracts rather
  than counted as generation wins;
- the newest context collectors expose previously missing local predecessor
  helpers.

Circular signs:

- repeated fresh-slice proof-lane/context passes did not move beyond `1/5`;
- several paid runs converted compile failures into clean declines, which is
  useful but not proof progress;
- Catalan/LGV and NPartition-like problems still need actual Lean proof
  synthesis or manual lemma work after context improves.

Speed judgment: local engineering and diagnostics are fast enough to produce
meaningful artifacts every 15-45 minutes. Acceptance progress is slower:
roughly eight hours moved from a dev-panel `2/5` state to dev-panel `3/5`, but
fresh held-out-like acceptance remained `1/5`. The bottleneck is no longer
mostly OpenRouter spend. It is deciding when a unit has enough context and must
switch to proof synthesis or stronger-model/manual proof work.

## Blockers

- Fresh theorem units often need same-unit helper lemmas or project bridge
  lemmas that are visible in source intent but absent from prompt-ready context.
- DeepSeek V4 Flash with reasoning disabled is cheap and disciplined, but it
  under-solves harder Lean proofs.
- Some prompt rules are authored from the same units being re-evaluated.
- There is no single compact ledger yet that joins run id, unit ids, model,
  cost, wall time, context counts, failure class, and semantic outcome.
- The proof-lane acceptance path is still artifact-heavy; it can take many
  commands to answer "did this change improve the panel?"
- Partial-proof diagnostics reveal concrete Lean errors, but they do not yet
  automatically route to a Mathlib search task, project-dependency collector,
  stronger-model proof lane, or manual Lean lemma task.

## Recommendations

1. Freeze two panels now: one contaminated dev panel and one reserve fresh
   panel. A prompt rule can be authored on the dev panel, but must be scored on
   the reserve panel before being called generic.

2. Add a run ledger. One row per run should include timestamp, artifact root,
   units, stage, model, reasoning setting, prompt/completion/reasoning tokens,
   cost, elapsed time, checked context count, failed context count, compile
   count, semantic count, and failure classes.

3. Run the ablation matrix above before more one-off proof-lane retries. Keep
   fixed selector payloads when comparing generation/proof models.

4. Add a controller stop rule: after two attempts on the same unit with the same
   failure class and enough checked context, stop prompt/context tuning and
   route to proof synthesis, manual Lean lemma development, or stronger-model
   ablation.

5. Keep DeepSeek V4 Flash as the cheap default for selector/context stages, but
   do not assume it is the right proof-lane model. Test DeepSeek V4 Pro and Kimi
   K2.6 on identical proof-lane tasks.

6. Treat clean declines as a separate metric. They are a quality improvement
   over invalid code, but they should not be counted as proof progress.

7. Preserve the target-hidden contract exactly. Continue using gold metadata
   only for post-hoc checking and same-source exclusion filters, and keep
   leakage scans in every task-building step.

This reinforces the main bottom line. The project is moving toward a more

evidence supports: cheap context selection, recoverable model calls, useful
declines, and better hidden-target hygiene. It does not yet support: robust
fallback proof synthesis across fresh theorem units.

The optimistic path is still plausible in shape: Flash-class models select and

hard units, and manual Lean work is reserved for a small tail. But the observed
fresh-slice acceptance is too low to claim that the tail is small. A credible
next milestone is not whole-book scaling; it is a frozen dev+fresh ablation
showing materially higher fresh-panel compile/semantic coverage under the same
target-hidden policy.

## Addendum: Corrected Top-Line Probe

The first GPT-5.5 swap was not a proper top-line test: it used the provider
default reasoning path and an 8k completion budget. After correction, the

context/lemmas or iterative proof repair, not only larger completion budgets.

## Addendum: Full Open-Model Proof-Lane Ablation

The same fresh five-unit proof-lane task dir was then run with open models on

- DeepSeek V4 Pro, no reasoning, 8k cap: `$0.016758781`, `0/5` compile,
  `0/5` semantic. It mostly produced clean declines; unit 002 failed on a

This makes the open-model picture sharper. Kimi is currently the strongest cheap
open proof-lane model on this slice, but it still only recovers the easy theorem.
DeepSeek V4 Pro without reasoning is not an improvement over the Flash baseline
for proof synthesis. The remaining hard units now need a different lane:
error-aware repair, lemma/context synthesis, or stronger reasoning settings,
rather than another same-prompt no-reasoning retry.

Process fix: the Kimi retry exposed an accounting gap. The first unit-003 call
spent money and returned invalid truncated JSON; the resume retry overwrote that
batch before attempt archiving existed. A durable `retry-cost-audit.json` now
preserves that cost, the run ledger includes manual retry adjustments, and the
proof-lane generator now archives failed paid attempts before retrying.
