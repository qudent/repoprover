For each unit, the selector prompt arranges information in this order:

1. `source_unit`: public source id, environment, path, line range, labels,
   referenced labels, part markers, parse warnings, and verbatim LaTeX source.
2. `source_context_candidates`: explicit references plus bounded previous
   same-file source units.
3. `previous_source_context`: previous source text only.
4. `prior_project_context`: declarations from earlier or explicitly referenced
   source units, not the selected unit.
5. `local_file_context_candidates`: namespace/open/style context inferred only
   from safe prior project declarations.
6. `local_file_predecessor_declarations`: same Lean file declarations before
   the selected unit placement line, with selected-unit aligned/referencing
   declarations withheld.
7. `benchmark_policy`: target Lean unavailable, posthoc alignment hidden, local
   predecessor context uses benchmark placement metadata only as a cut line.

Justification: this is the central target-blind planning stage. Its purpose is
to avoid giving the generator all context indiscriminately while also avoiding
single-declaration rows that are too narrow for theorem-like LaTeX units.

## Benchmark-Honesty And Cheating Review

No direct target Lean declaration names, statements, or proofs are intentionally
included in selector, generator, repair-context, or repair prompts. The prompts
state repeatedly that hidden aligned declarations are withheld. Gold comparison
and semantic coverage are post-hoc.

Honest mechanisms:

- `target_lean_available_to_selector/generator/repair` is false in payload
  policy blocks.
- Prior project declarations are included only when they belong to earlier or
  explicitly referenced source units.
- Selected-unit aligned and referencing declarations are withheld from local
  predecessor context.
- Failed exact Mathlib identifiers are redacted before generation.
- Unchecked fallback candidates are search hints, not facts.
- `raw_invalid_generation_output` is explicitly labeled scratchpad-only.
- Verifier now filters inferred imports/opens that would expose hidden target
  modules or namespaces.
- Contract normalizer prevents placeholder skeletons from counting as generated
  proof attempts.

Main honesty risks:

1. `local_file_predecessor_declarations` uses post-hoc target file/line
   placement metadata as a cut line. It omits the target declaration, but it
   still uses oracle placement. This is acceptable for diagnostic research only
   if reported as such. A production benchmark needs a target-blind source-to-
   file placement method.
2. `docs/npartition-helper-route-diagnostic-2026-05-06.md` is explicitly
   post-hoc and says it must not be model-facing. The prompt changes derived
   from it are generic, but they were learned while debugging the NPartition
   datapoint. Counting a later NPartition success as a clean held-out benchmark
   result after these prompt edits would be questionable.
3. Some prompt rules now name concrete Mathlib facts such as
   `List.ofFn_get`, `List.sum_ofFn`, `Finset.sum_union`, `Finset.sum_eq_zero`,
   and `Finset.sum_map` as examples for finite-sum conversion. These are framed
   generically, but the exact examples came from the NPartition route. This is
   not hidden target-name leakage, but it is datapoint-informed prompt tuning.
   It should be evaluated on holdout theorem units before being counted as a
   general benchmark improvement.
4. Fresh helper name hints like `fromPartition` appear in model outputs and
   checked repair context. They are not scripted hidden gold names, but they are
   datapoint-specific. The rule says fresh descriptive names are allowed; still,
   scorekeeping should not reward name overlap here.

Bottom line: the current process is mostly honest for research iteration, but
not clean enough to claim a held-out benchmark win on NPartition if it later
succeeds using prompt lessons learned from NPartition's gold/local source
diagnostic. Treat NPartition as a development case; validate the generic
prompt/routing lessons on fresh held-out theorem units.
