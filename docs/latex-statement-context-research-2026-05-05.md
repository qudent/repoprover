# LaTeX Statement Context Research - 2026-05-05

## Scope

This note records the current evidence for moving from one hidden Lean
declaration per benchmark row to one LaTeX theorem/definition/etc. per source
unit. It distinguishes three things that should not be conflated:

- source-unit extraction from LaTeX;
- source/project/Mathlib context-pack selection;
- Lean dependency accounting from checked/elaborated declarations.

## LaTeX Source Dependencies

Each source-unit row includes:

- `source_unit.referenced_labels`: labels explicitly referenced in the LaTeX
  environment using `\ref`, `\eqref`, `\cref`, etc.;
- `context_candidates.referenced_source_units`: resolved source units for those
  references when the referenced label is also a parsed theorem-like unit;
- `context_candidates.previous_same_file_source_units`: nearby earlier labeled
  source units in the same file.

This is a useful source-context graph, but it is not a proof-dependency oracle.
Explicit source references are high-precision for "the text points here"; they
are incomplete for implicit mathematical dependencies, notation conventions,
definitions introduced in prose, and proof dependencies that only appear in the
Lean development.

Known source parsing caveat: one `LGV1.tex` unit has a mismatched
`\begin{definition}` / `\end{example}` pair in the source. The dataset keeps the
unit and records `parse_warnings` rather than dropping it.

## Open-Model Context-Pack Evidence

The older live context-selector probes are on declaration-level rows. They are
still useful because they test whether models can produce tight Mathlib/project
context without seeing the hidden Lean statement. A first LaTeX-unit selector
smoke now exists too; it tests the new theorem-level prompt without generation
or Lean verification.

Best result:

- model: `deepseek/deepseek-v4-flash`
- mode: no reasoning parameter
- batch size: 2
- result: valid JSON, hydrated Mathlib snippets, zero target-name payload leaks
- cost: `$0.0025886`

Failed or weak modes:

- `moonshotai/kimi-k2.6`, low/no reasoning: mostly hidden reasoning, no usable
  content before output cap.
- `deepseek/deepseek-v4-flash` with reasoning: hidden reasoning consumed the
  output budget.
- `google/gemini-2.5-flash`: text output, but truncated/invalid JSON with the
  current verbose schema.
- batch size 4: often truncates with the current schema.

What worked:

- The selector can ask for plausible Mathlib facts such as `Matrix.det_diagonal`
  and permutation-sign lemmas.
- Hydration can resolve requested Mathlib names into real local source snippets.
- Project context was more decisive than pure Mathlib lookup in successful
  verification cases: `alternant_swap_of_ne` reused imported
  `alternant_swap`, and `isInverse_unique` reused imported `inverse_unique`.

What did not work yet:

- Generation/verification remains low: project-context probes reached `1/2` and
  `1/3` on small slices.
- Failures were often source-shape/planning failures, not only Mathlib recall:
  broad source labels caused bundled conjunctions or over-broad theorem shapes.
- Determinant examples still fail when the generator invents unavailable helper
  theorem names or malformed proof syntax.

Conclusion: cheap open selectors can generate useful context packs, but only
when the task is tightly scoped and previous-project context is made explicit.
For theorem-level units, the selector needs to output an ordered declaration
plan first, then request Mathlib/project context per planned declaration.

## Lean Dependency Accounting

There are two dependency views:

- `scripts/analyze_lean_usage_static.py`: fast text scan over source. It sees
  explicit names but misses notation expansion, typeclass-generated terms, and
  tactic-produced proof terms.
- `scripts/analyze_lean_dependencies_elaborated.py` plus
  `scripts/lean_dependency_scan.lean`: Lean-tooling scan over checked
  `ConstantInfo`, recording direct resolved constants from elaborated
  declarations.

The elaborated scan writes one row per project constant in the compiled
environment, including private/generated proof helper constants. These rows are
lower-level than LaTeX units and lower-level than the old 5,684 source
declaration records. They answer "what constants does this checked Lean
constant directly mention?", not "what source theorem is complete?"

The full elaborated scan is intentionally heavy on this 8 GB machine. It should
be treated as background evidence for dependency accounting, not as a blocker
for source-unit extraction.

Current completed scan:

- raw unique project constant rows: `16,485`
- record summary input: `docs/latex-statement-gold-candidates.jsonl`
- record schema: `repoprover.latex_statement_unit.v1`
- records summarized: `114`
- missing aligned declarations in raw scan: `10`
- direct unique constants across those rows: `1,456` Mathlib, `852` project,
  `315` other Lean constants
- aligned Lean declarations per LaTeX unit: median `2`, p90 `8`, p95 `12`,
  max `29`
- direct Mathlib constants per LaTeX unit: median `44`, p90 `152`, p95 `169`,
  max `444`
- direct project constants per LaTeX unit: median `5`, p90 `26`, p95 `34`,
  max `85`

These counts are much larger than the static source mention counts because they
include elaborated proof terms, generated helper constants, and non-obvious
dependencies introduced by notation, tactics, and typeclass inference. They are
dependency-accounting evidence, not a recommendation to put hundreds of raw
constants into each prompt.
